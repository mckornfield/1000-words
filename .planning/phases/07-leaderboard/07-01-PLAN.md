  <action>
    Create supabase/migrations/20260624000000_leaderboard_rpc.sql.

    Write one SQL function: public.get_leaderboard(n integer default 50).

    The function must:
    - Be declared language sql, security definer, set search_path = '' (matching the increment_xp pattern from 20260622000000_user_state_tables.sql line 91-93)
    - Return a TABLE with columns: user_id uuid, display_name text, xp integer, level integer, achievement_count bigint, rank_value bigint, border_item_id text, badge_item_id text, avatar_item_id text
    - SELECT from public.profiles p LEFT JOIN public.user_achievements ua ON ua.user_id = p.user_id LEFT JOIN public.user_equipped ue ON ue.user_id = p.user_id
    - Compute level as (floor(p.xp::float / 250)::int + 1)
    - Compute achievement_count as count(ua.achievement_id)
    - Compute rank_value as level * achievement_count (using the level expression above)
    - Pivot equipped cosmetics via MAX(CASE WHEN ue.slot = 'profile_border' THEN ue.item_id END) as border_item_id, MAX(CASE WHEN ue.slot = 'profile_accent' THEN ue.item_id END) as badge_item_id, MAX(CASE WHEN ue.slot = 'profile_picture' THEN ue.item_id END) as avatar_item_id
    - GROUP BY p.user_id, p.display_name, p.xp
    - ORDER BY rank_value DESC, p.xp DESC
    - LIMIT n
    - Clamp input: add CHECK that n <= 100 in the function body using LEAST(n, 100) rather than a constraint (simpler, no raise needed)

    After the function definition, add: grant execute on function public.get_leaderboard(integer) to authenticated;

    Also add a second function: public.get_current_user_rank(uid uuid) returning the same TABLE columns as get_leaderboard but with a WHERE p.user_id = uid clause and no LIMIT. Same security definer / search_path pattern. Grant execute on this function to authenticated as well.

    Do NOT add new RLS SELECT policies on the underlying tables — the SECURITY DEFINER function bypasses RLS at execution time. Adding permissive policies would unnecessarily widen cross-user data access.

    Include a comment block at the top of the file explaining: "Leaderboard RPC. SECURITY DEFINER allows cross-user aggregation without widening RLS on base tables."
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → Supabase RPC | Authenticated user calls get_leaderboard(n) — n is user-controlled |
| SECURITY DEFINER function → base tables | Function executes as owner, bypasses per-user RLS — must not return PII |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Information Disclosure | get_leaderboard RPC | mitigate | Function returns only display_name, xp, level, achievement_count, rank_value, and cosmetic item IDs — no email, phone, or auth UID. user_id is returned but is a UUID, not PII in this context. Reviewed in Task 1. |
| T-07-02 | Tampering | n parameter to get_leaderboard | mitigate | Parameter is integer type in SQL — no injection possible. LEAST(n, 100) caps at 100 rows, enforced at DB level per security contribution requirement. |
| T-07-03 | Tampering | rank_value computation | accept | Rank computed server-side in SQL; client only renders returned rows. Client-side rank manipulation would only affect display, not stored state. No stored rank column. |
| T-07-04 | Information Disclosure | SECURITY DEFINER RLS bypass | mitigate | Function uses set search_path = '' (matching increment_xp pattern). Function only SELECTs from profiles/user_achievements/user_equipped — no INSERT/UPDATE/DELETE. Declared stable via language sql (implicitly read-only). |
</threat_model>
